Product Spec · App 02

Aurë

A fragrance-intelligence & scent-wardrobe app — fragrance treated as a wearable identity system, not a perfume catalog.

BetaFlutterSupabaseMotion-first
What it is

The 10-second version

An app that maps your fragrances like a wardrobe: what you wear, when, the mood and "aura" each scent gives, layering and styling logic, wear history, and context-aware suggestions. It feels like a designed emotional interface — atmospheric, editorial, sensory.

Why I built it

The spark

Scent is one of the most personal things we own and one of the hardest to track. Existing apps are marketplaces. I wanted the opposite — a private, beautiful place to understand and style your own collection.

The problem it solved for me

A real app, solo

I had a deep product vision and no engineering team. AI let me take it from a React Native prototype all the way to a modular Flutter app with a real architecture — something I could never have hand-built alone.

The problem it solves for a user

Scent, organized & expressive

Fragrance lovers juggle dozens of bottles with no memory layer. Aurë gives them a wardrobe, a wear log, and recommendations tied to mood and context — turning a messy shelf into a styling system.

How it works

Under the hood

A feature-modular Flutter app: a Galaxy view (constellation + grid) to browse the collection, a Fragrance Detail screen with collapsing hero and memory/community states, City Scent (context-aware suggestions), and a Wear Log archive. Shared theme + motion tokens keep every screen feeling like one designed object. Riverpod for state, go_router for routing, Supabase as the backend target.

Where it's going · how it monetizes

The business model

  • Freemium — the wardrobe + wear log free; styling intelligence, layering logic, and context suggestions behind a subscription.
  • Affiliate — "complete the wardrobe" recommendations link to retailers; revenue on discovery without becoming a marketplace.
  • Premium drops — collabs with niche houses for limited in-app content.

Next: finish the Wear Log archive, wire Supabase, and ship a TestFlight beta to fragrance communities.

What I learned building it

How I directed the AI

Old app = spec, new app = source of truth

I was migrating from React Native to Flutter. The rule I gave the agent: treat the old prototype as a reference spec only and the Flutter project as the single source of truth. That one constraint stopped it from blindly porting code and let it rebuild cleanly while preserving the product logic.

How I directed the AI

Phase it, and demand a report

I broke the rebuild into Phases 0→7 (stabilize → foundation → features → data → backend) and gave the agent a fixed output format every pass: Summary · Files changed · What's working · Blockers · Next step. Phasing kept it from overbuilding; the report format made every session resumable.

What broke & how I fixed it

AI leaves a mess behind

Iterative AI edits pile up duplicate app shells, dead routes, and stale scaffolding. The fix was a dedicated Phase 0 cleanup plus a standing rule — "the analyzer must pass cleanly after each phase, no speculative broad rewrites." Discipline beats vibes.

The automation win

A reusable "continue" prompt

Because the plan was phased and the output format fixed, I wrote one master prompt that resumes the build from wherever it left off. Open a fresh session, paste it, and the agent picks up the next phase — no re-explaining the project each time.
